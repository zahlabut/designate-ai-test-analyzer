Agentic AI architecture

How CrewAI + Ollama are used in main.py — agents, tasks, and code structure.

Mirrors the module docstring at the top of main.py.

Designate E2E Test Investigator is an agentic-AI CLI for debugging failing Designate Tempest tests on DevStack. It combines deterministic orchestration (shell, stestr, journalctl, source parsing) with LLM reasoning (CrewAI + local Ollama) in a fixed pipeline.

Agentic AI design

This is agentic because a CrewAI Agent performs open-ended reasoning at two points: test intent (Stage 1) and root-cause verdict (Stage 3). It is not a multi-agent swarm — one agent, two tasks, no delegation, no tool loop.

ComponentDetails
Framework CrewAI — Agent / Task / Crew
LLM backend Ollama (OpenAI-compatible API at 127.0.0.1:11434); model chosen at startup (e.g. ollama/llama3.2:1b)
Config conf.ini sets OPENAI_API_BASE and OPENAI_API_KEY for CrewAI internally

Single agent — “Designate Expert Architect” (get_analyst())

PropertyValue
roleDesignate Expert Architect
goalExplain and troubleshoot Designate E2E tests with clear, accurate technical summaries
backstoryDevStack / Designate architecture context (SYSTEM_CONTEXT — API, Central, Producer, Worker, mDNS)
toolsnone (tools=[]) — all context is injected into each Task description
allow_delegationFalse — no sub-agents or hand-offs

Created lazily after model selection (setup_ollama_model), cached in _analyst. reset_analyst() clears the cache when the model changes.

Tasks (same agent, one per LLM stage)

Stage 1 — run_stage_logic_discovery LLM

  • Input (deterministic): load_test_source_bundle() reads the test method and helper methods from designate-tempest-plugin source.
  • Task description: full source + instructions to explain setup, DNS checks, waits, and expected Designate behaviour in plain prose.
  • Output: test-intent summary (cyan panel).

Stage 3 — run_stage_root_cause (FAIL only) LLM

  • Input (deterministic): build_log_evidence_report() — Tempest DNS config, traceback, run log excerpts, per-service journal summaries (designate-api, central, worker, …).
  • Task description: test intent + evidence + structured questions (client vs backend, which services errored, most likely root cause).
  • Output: root-cause verdict (magenta panel).

Execution: run_crew_task() → Crew(agents=[get_analyst()], tasks=[task]).kickoff()

Why no CrewAI tools? Early versions used a read_source tool; the LLM often returned JSON tool-call syntax instead of prose. Source and logs are now pre-loaded deterministically and pasted into the Task — the agent only writes analysis.

Hybrid logic: Stages 1 and 3 are LLM-assisted; Stage 2 is pure subprocess (stestr run --serial). The closing run summary mixes deterministic hints (failure_context_hint, backend_logs_hint) with extracted LLM text (summarize_intent, summarize_root_cause).

Pipeline

  Setup          Stage 1              Stage 2              Stage 3
  ─────          ───────              ───────              ───────
  Ollama ✓   →   Analyze logic   →   Run test   →   Root cause
  tempest ✓      (test source)      (stestr)            (journalctl)
                   cyan · LLM          yellow              magenta · LLM
                                        │
                                        ├─ PASS  → done
                                        ├─ SKIP  → done
                                        └─ FAIL  → Stage 3
  1. Setup — Ollama health + model pick, tempest.conf check, grep/list tests
  2. Stage 1 — load test source → CrewAI Task → test intent
  3. Stage 2 — stestr run → PASS / FAIL / SKIP (no LLM)
  4. Stage 3 — on FAIL: journalctl → evidence report → CrewAI Task → verdict
  5. Summary — brief recap + artifacts under /opt/stack/agent_runs/run_<ts>/
